공식문서를 읽을 수 있는 힘

제로베이스 김데레사님의 HTML / CSS 수업 때 공식문서의 힘을 엿볼수 있었습니다.

나는 공식문서를 읽을 수 있는가?

  • [ ● ] 공식문서가 있는 페이지를 방문할 수 있다.
  • [ ● ] 영어로 된 공식문서를 해석할 수 있다.
  • [ ● ] 공식문서를 읽고 이해할 수 있다.
  • [ ▲ ] 내 코드에 적용할 수 있다.
  • [ x ] 공식문서 소개의 목적이나 목표에 맞게 적용할 수 있다.

공식문서가 있는 페이지를 방문할 수 있다.

저는 공식문서가 있는지 검색할 수 있습니다. 예를 들어 HTML 태그에 대한 공식문서가 필요하다면, Whatwg 표준문서를 참조합니다. 또는 w3 공식 사이트를 방문합니다. 2가지 중 Whatwg 표준문서를 우선적으로 검색합니다. 그 이유는 w3의 표준화 속도가 느려 유명 브라우저 벤더들이 다른 문서를 작성하니, 이 단체가 whatwg입니다.

영어로 된 공식문서를 해석할 수 있다.

대부분의 라이브러리나 프레임워크는 북미나, 유럽권에서 배포됩니다. 최신의 정보는 영어로 된 문서가 많습니다. 한글 업데이트가 매순간 이뤄진다면 좋겠지만 쉬운 일이 아닙니다. 그래서 우리는 영어로 문서를 읽고 해당 기술을 이해할 능력이 요구됩니다.

공식문서를 읽고 이해할 수 있다.

설명이 불친절한 문서는 어려울 수 있지만, 커뮤니티가 크거나 짜임새가 있는 문서라면 이해도도 높아집니다. 모든 글을 읽지 않아도 흐름으로 내가 찾고자 하는 부분을 알 수 있습니다. 즉 내가 이해되지않은 부분은 자세히 읽고, 알 수 있는 부분은 융통성을 발휘할 수 있습니다. 이는 시간의 절약과, 생산성 향상에 도움을 줍니다. 예를 들어 webpack 처럼 내가 깊거나 자세히 사용법이 알고 싶은 빌드 툴 또는 라이브러리등이 있다면, Guides와 Documentation을 모두 참조하며 읽어보고 이해하겠지만, Hexo처럼 그때 그때 필요한 기능을 검색해 사용할 도구들이 있습니다. 때와 상황에 따라 문서를 읽는 방법을 터득했습니다.

내 코드에 적용할 수 있다.

도구의 목적과 동작 순서등을 이해한다면 충분히 코드에 적용할 수 있습니다. 예를 들어 todos application을 통해 Javascript를 이해하는 도중에 webpack을 적용한다거나, 스타일시트에 변화를 줘야할 때 적절한 스크립트나, 파일구조, 순서등을 유려하게 수정할 수 있습니다.

공식문서 소개의 목적이나 목표에 맞게 적용할 수 있다.

앞서 말했던 코드의 적용이 완전하지 않습니다. 도구들의 목적이나 목표룰 파악하는 연습이 충분하지 않아 미흡합니다. 미흡한 점을 발견했으니, 보충할 방법이 있습니다. 디자인, 아키텍처, 시스템 등 전박적 소프트웨어 기술에 대한 이해도를 높이면, 더욱 더 제가 읽은 공식문서의 힘은 강해집니다.

공식문서의 힘을 통해 무엇을 얻을 수 있나?

  • 설득력
  • 범용성

나의 의견에 전문성 또는 정확성을 공식문서라는 근거로 높일 수 있습니다.

내가 작성한 문서를 다른 사람이 참조하기 좋다.

내가 공식문서의 일반적 구조를 이해하고 알고 있다면, 내가 작성한 문서들을 다른 사람이 읽어도 금방 이해할 수 있도록 작성할 수 있어집니다. 팀 프로젝트를 진행하면서도 내가 공부하고 진행한 환경설정에 대해 문서를 프로젝트 막바지에 표준화된 공식문서를 작성할 수 있다면, 시간을 절약할 수 있습니다.

How & Why use github?

협업할 수 있는 능력을 가지자 / 부제: Github을 쓸줄 알아야지

Webpack과 commit을 연습중

Github이 무엇인가

서비스이다. GIT을 활용한 버전관리를 온라인상에서 저장할 수 있도록 해주는 서비스입니다. 서비스를 강조한 이유는 Github과 Git은 엄연히 다르다는 것을 기억해두기 위함입니다.

내가 그동안 사용한 Github

  1. git add . && git commit -m "Blah Blah" && git push

    와 github repository에 성공적으로 올라갔다. Github 써보았다.

  2. git add를 선택적으로 그리고 git commit -m 보다 git commit을 통한 커밋 저장이 왜 중요한가?

  • git은 버전관리 또는 형상관리를 위한 도구입니다. 하나의 서비스를 만드는 회사나, 팀이 Github으로 버전을 관리한다는 것은 나 이외에 다른 사람과 커밋을 공유한다는 뜻입니다. 그래서 정해진 커밋 컨벤션에 맞게 커밋을 작성하고 그 커밋은 동작하는 하나의 묶을으로 add하여 의미있는 파일들을 Repository에 push해야 합니다.
  • 아직 개발자로 경험해본적도 없는 제가 어떻게 이를 느꼈을까요? Conventional Commits 많은 개발자들은 표준화된 것을 좋아합니다. 그래서 널리 알려진 컨벤션들을 활용하는 회사도 많습니다. 오픈소스를 읽어도 git clone 뿐이였습니다. 하지만 컨벤션을 지킨 이슈와 풀리퀘스트 등을 관찰할 수 있게 되었습니다.
  • 대표적 예로 toss/slash 작성일(11-08) 기준으로 토스에서 오픈소스를 두어달 전에 오픈했습니다. 오픈소스에 이슈들과 풀리퀘스트를 읽고 이것이 BUG이고 FEAT 이런 컨벤션이 지켜져 올라가는 것을 보고 저 또한 일면식 없는 사람들의 메세지를 이해하고 읽을 수 있었습니다.

    toss/slash는 타입스크립트로 이루어진 패키지 모음입니다. 읽을 수 있지만 컨트리뷰트를 아직 못하는 것이 정말 아쉽습니다. 아직 ES6+와 Webpack에 집중하고 있습니다. 그래서 타입스크립트을 잘 알지못합니다.

How to write code?

Posted on 2022-11-07 Edited on 2022-11-17

가독성이 좋은 코드

Deep Dive를 통해서

“결국에 이 책은 모두 가독성에 대한 내용이에요.” - 이웅모 (Deep Dive 저자)

용어

추상적 단어나 독자적인 이해로 만든 용어는 같이 일하는 사이에서 모호함을 줄 수 있습니다. 약속된 혹은 정확한 용어로 표현함으로써, 보다 소통이 원할하게 이뤄질 수 있습니다. 예를 들어서, 생성자 함수와 일반 함수라는 표현을 독자적 추상적 단어로 동료에게 표현한다면 이 용어에 대한 추가설명등이 필요해집니다. 불필요한 일이 발생하는 것입니다.

네이밍

Deep Dive를 읽으면 식별자의 네이밍 규칙등이 나오는데, 그런 규칙들과 동시에 함수나 변수의 역할을 명시적으로 직관적으로 알 수 있도록 하는 네이밍을 의미합니다. 나를 위해서 그리고 나와 일할 동료를 위해 시간이 지나도 처음 읽어도 이름만으로도 이해할 수 있다면 좋은 네이밍입니다. 예를 들어, $를 변수명에 앞에 위치시켜 Node 객체를 할당받은 변수를 나타낸다거나, const todos = [] 객체 리터럴로 작성해서 null이나 undefined로 할당했을 때보다 명시적으로 todos라는 변수에는 배열이 온다는 것을 나타낼 수도 있습니다.

사람이 이해할수 있도록

for(let i = 0 ; i < 10 ; i++)은 자료구조와 알고리즘 혹은 코딩테스트를 준비한 사람이라면 모두 쉽게 읽을 수 있는 코드입니다. ‘i를 1씩 증가시켜 10번의 같은 문을 반복해줘’의 의미를 가진 문입니다. 하지만 그냥 읽었을 때는 문법을 모르는 사람이라면 이해하기 어렵습니다. for(let i = 0 ; i < 10 ; i += 1) 세번째 조건인 증감을 += 할당연산자로 조금 더 사람이 읽을 수 있도록 작성해봤습니다. 하지만 여전히 사람이 이해하기에는 쉬운 코드는 아닙니다. 그래서 영단어 조합으로 조금 더 이해도를 높일 수 있도록 해보겠습니다. function addOneTenTimes() ‘1을 열번 더해줘’라는 의미에 이름으로 함수를 작성했습니다. 조금 더 이해도가 올라갔지만, 의미는 퇴색되보입니다. 이름을 문을 열번 반복한다는 의미에 맞게 조금 변경시켜보겠습니다. function repeatTenTimes() ‘열 번 반복해줘’라는 의미가 전달되도록 이름을 지었습니다.

명령형 VS 선언형

프론트엔드 개발자가 되고 싶다면, 성능을 무시할 수 없습니다. 그래도 Deep Dive와 마틴 파울러의 리팩토링에서는 성능에 문제가 된다면 이후에 살펴봐도 된다고 서술합니다. 성능에 영향을 끼칠 정도로 무거운 코드가 있다면, 그 자체가 문제일 수 있으며 작성된 코드는 모수가 충분히 적다면 브라우저 벤더들이 제작한 브라우저 환경에서 무리가 있을 확률은 적기 때문입니다. 고로 앞서 업근된 for문처럼 기계에게 작동할 문들을 하나씩 알려주는 방식보다는 선언적으로 사람이 이해할 수 있도록 네이밍과 구조를 작성하고, 기계는 그 네이밍이나 구조안에 동작을 이해하여 실행할 수 있도록 작성하는 것을 우선으로 생각합니다.

자바스크립트다움?

“자바스크립트는 자바스크립트답게 작성하세요.”
이웅모님과 함께 자바스크립트를 배우고 공부를 할때 제일 집중했으며 이해가 어려웠던 말이었습니다.

첫 힌트를 얻은 것은 19장 프로토타입의 서두에 나오는 멀티 패러다임 프로그래밍 언어다 입니다. 자바스크립트는 프로토타입으로 객체지향 프로그래밍을 구현할 수 있습니다. 앞선 가독성 관점으로 생각해서 자바스크립트를 여러 패러다임의 장점을 이용해서 가독성 좋게 명시적으로 코드를 작설할 수 있다고 생각합니다.(아직 확실히 이해했다거나 개인적 생각정리가 완료되지는 않았습니다.)
