가독성이 좋은 코드

Deep Dive를 통해서

“결국에 이 책은 모두 가독성에 대한 내용이에요.” - 이웅모 (Deep Dive 저자)

용어

추상적 단어나 독자적인 이해로 만든 용어는 같이 일하는 사이에서 모호함을 줄 수 있습니다. 약속된 혹은 정확한 용어로 표현함으로써, 보다 소통이 원할하게 이뤄질 수 있습니다. 예를 들어서, 생성자 함수와 일반 함수라는 표현을 독자적 추상적 단어로 동료에게 표현한다면 이 용어에 대한 추가설명등이 필요해집니다. 불필요한 일이 발생하는 것입니다.

네이밍

Deep Dive를 읽으면 식별자의 네이밍 규칙등이 나오는데, 그런 규칙들과 동시에 함수나 변수의 역할을 명시적으로 직관적으로 알 수 있도록 하는 네이밍을 의미합니다. 나를 위해서 그리고 나와 일할 동료를 위해 시간이 지나도 처음 읽어도 이름만으로도 이해할 수 있다면 좋은 네이밍입니다. 예를 들어, $를 변수명에 앞에 위치시켜 Node 객체를 할당받은 변수를 나타낸다거나, const todos = [] 객체 리터럴로 작성해서 null이나 undefined로 할당했을 때보다 명시적으로 todos라는 변수에는 배열이 온다는 것을 나타낼 수도 있습니다.

사람이 이해할수 있도록

for(let i = 0 ; i < 10 ; i++)은 자료구조와 알고리즘 혹은 코딩테스트를 준비한 사람이라면 모두 쉽게 읽을 수 있는 코드입니다. ‘i를 1씩 증가시켜 10번의 같은 문을 반복해줘’의 의미를 가진 문입니다. 하지만 그냥 읽었을 때는 문법을 모르는 사람이라면 이해하기 어렵습니다. for(let i = 0 ; i < 10 ; i += 1) 세번째 조건인 증감을 += 할당연산자로 조금 더 사람이 읽을 수 있도록 작성해봤습니다. 하지만 여전히 사람이 이해하기에는 쉬운 코드는 아닙니다. 그래서 영단어 조합으로 조금 더 이해도를 높일 수 있도록 해보겠습니다. function addOneTenTimes() ‘1을 열번 더해줘’라는 의미에 이름으로 함수를 작성했습니다. 조금 더 이해도가 올라갔지만, 의미는 퇴색되보입니다. 이름을 문을 열번 반복한다는 의미에 맞게 조금 변경시켜보겠습니다. function repeatTenTimes() ‘열 번 반복해줘’라는 의미가 전달되도록 이름을 지었습니다.

명령형 VS 선언형

프론트엔드 개발자가 되고 싶다면, 성능을 무시할 수 없습니다. 그래도 Deep Dive와 마틴 파울러의 리팩토링에서는 성능에 문제가 된다면 이후에 살펴봐도 된다고 서술합니다. 성능에 영향을 끼칠 정도로 무거운 코드가 있다면, 그 자체가 문제일 수 있으며 작성된 코드는 모수가 충분히 적다면 브라우저 벤더들이 제작한 브라우저 환경에서 무리가 있을 확률은 적기 때문입니다. 고로 앞서 업근된 for문처럼 기계에게 작동할 문들을 하나씩 알려주는 방식보다는 선언적으로 사람이 이해할 수 있도록 네이밍과 구조를 작성하고, 기계는 그 네이밍이나 구조안에 동작을 이해하여 실행할 수 있도록 작성하는 것을 우선으로 생각합니다.

자바스크립트다움?

“자바스크립트는 자바스크립트답게 작성하세요.”
이웅모님과 함께 자바스크립트를 배우고 공부를 할때 제일 집중했으며 이해가 어려웠던 말이었습니다.

첫 힌트를 얻은 것은 19장 프로토타입의 서두에 나오는 멀티 패러다임 프로그래밍 언어다 입니다. 자바스크립트는 프로토타입으로 객체지향 프로그래밍을 구현할 수 있습니다. 앞선 가독성 관점으로 생각해서 자바스크립트를 여러 패러다임의 장점을 이용해서 가독성 좋게 명시적으로 코드를 작설할 수 있다고 생각합니다.(아직 확실히 이해했다거나 개인적 생각정리가 완료되지는 않았습니다.)
